ebase
From past to present.
Google Gründer Larry Page und Sergey Brin treten ab.

2000

Long before ebase,

anything starts with an Excel Macro...

Eigentlich wollte ich eine Mathematisch/Numerische Arbeit schreiben. Und nicht bei Edscha anfangen... oder höchstens 5 Jahre bleiben.

2001

The very first ebase version, known as "VDM1" and build as client/server web application, is published.

2002

New requirements lead to the creation of VDM2.

It's a partitial rewrite of VDM1.

2003

Another iteration

It turns out that the VDM2 architecture does not fit the increasing demands anymore.

VDM3 is born

Which means the second rewrite, now from scratch.

VDM3 wurde maßgeblich mit der Unterstützung von Hr. Heinrichs voran getrieben.
But by still using the same technology stack.

Because in 2003 it's still unusual to run enterprise applications in the browser, it's worth to publish an article about it.

2006

Migration to PHP5, Apache2 and MySQL4.

From now on, we call it "ebase"!

2006

I am moving from the Engineering to the IT department.

ebase gets out of focus for a while.

2008

Because HTML4 is still not satisfactory, the so called "Rich Internet Client (RIA)" era arise.

2008

A movement away from the browser to RIAs and mobile apps starts.

Adobe, Microsoft und Oracle entwicklen RIA Technologien. Adobe setzt sich durch und besteht am längsten.

2009

We jump on this RIA train and leave the HTML4 world in favour of the powerful Flash/Flex platform.

ebase still runs in the browser, but by using the FlashPlayer plugin

2013

"ebase prototyping" is launched.

It needs three iterations to reach the primary goals.

2015

HTML5, JavaScript and so called "Single-Page-Apps" gains more tracktion.
It is such a boom, that every day a new JavaScript frontend framework or library come out.

We decide to stick with Flash/Flex as long as possible.

Flex is still ahead of HTML5.

Beeing a committer and PMC member of the Apache Flex project we can participate in shaping the Future of Flex.

2017

The current, fragmented, ebase system is outdated in any respect.

Technology, Architektur und Inhalt sind nicht mehr zeitgemäß.

We start the effort on ebase4.

Still by using Apache Flex but with targeting the Desktop only using Adobe AIR.

ebase4 ist nur als Desktop Version verfügbar. FlashPlayer wird Ende 2020 nicht mehr verfügbar sein!

In order to be prepared for the future, we join the Apache Royale team.

Apache Royale ist ein Schwesternprojekt von Apache Flex. Es folgt dem Apache Flex Konzept und verwendet die gleichen Sprachen. Die Idee ist, besetehende ebase Anwendungen mit so wenig Aufwand wie möglich nach HTML5 zu migrieren bzw. neu zu entwickeln.

We hire Yishay Weis from Israel and

Piotr Zarzycki from Poland

in order to help us to shape the future of ebase.

Beide sind Flex UND Royale core Entwickler. Bis Royale einsatzfähig ist, bleiben wir bei Flex. Apache Royale ist ein Schwesternprojekt von Apache Flex. Es folgt dem Apache Flex Konzept und verwendet die gleichen Sprachen. Die Idee ist, besetehende ebase Anwendungen mit so wenig Aufwand wie möglich nach HTML5 zu migrieren bzw. neu zu entwickeln.

2019

We give "Apache Royale" a try in order to check-out if it fits our needs.

Wir entwickeln eine Formular-Anwendung für die IT.

The shock: "As of June 2019, Adobe is transitioning ongoing platform support and feature development of AIR to HARMAN"

This marks the end of the Flex era for us.

We immedeately stop any effort regarding ebase4!

But we need to switch to another technology...

Unfortuntely, our "Apache Royale" test fails.

It turns out that Royale doesn't fit our needs.

But we have to make a decision...

My upcoming parental leave is a good opportunity to cool down and find a new direction ;-)

With Vue and ebase5 we are back to the browser and back to web development, which is great!

With ebase5 we break down the monolithic ebase structure.

This is an important step for the future of ebase.

Das bringt Investitionssicherheit für Edscha.

We introduce the ebase API.

The ebase5 client is just the default client which consumes the ebase API.

Any other devices, machines and applications will be able to consume the ebase API.

APIs
APIs prove to be key to digital transformation

An application programming interface (API) is a way for software components to communicate.

APIs exist since we have software.

APIs are becoming the digital reflection of an organization.
The reality of an IT landscape in large enterprises.
API Services can act as a Facade which unlocks the data!
ebase5 is just a minor small system, but it's based on APIs... by default!
